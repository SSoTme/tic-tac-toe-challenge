About the challenge

This website serves as a highly detailed specification for a game of Tic-Tac-Toe (example).  Every detail of what the game should be, and what it should do has been described, from the board, players, cells, strategies, etc.  What has not been detailed is how the game should be created.

What won’t change

There is only 1 constant, which will never change - this is a 3x3, traditional tic-tac-toe game.  This is not Connect 4 or Connect N.  3x3 Tic-Tac-Toe is the sandbox we’re playing in.  Other than that, pretty much everything’s up for grabs.

What will change

The GSheet which defines this specification includes two different types of cells - white and gray.  All white cells are subject to change, at any time, gray cells are not expected to change.

  • White Cells: If only values with a white background are changed, and your solution requires hand code changes in response, points will be subtracted.

  • Gray Cells:  If values with a gray background change, and no hand code changes are required to comply, additional points may be awarded to your solutions - because, hand code changes are expected if gray background values change.

The Challenge

Implement this specification in the language, platform and environment of your choice.  Use whatever tools you want.

Scoring

Solutions will be scored by first accumulating points based on how many of the features in the specification are implemented, with subtractions for any deviations from “the spec”,  as well as subtractions for each additional hour of time invested.

The 3x3 Tic-Tac-Toe Specification

TicTacToe is a game that is simple enough that once you learn how to play it, you can either tie or win every game.  The only way to lose (once you know how) is to make a mistake.

This makes it a well suited problem for an entry level programming exercise.  This documentation is broken up into a few different parts.  Explore the Additional Resources for a complete list, but at a high level it works like this.

What this specification doesn’t say…

This specification makes no mention of any of the following:

  • The “kind” of app? (web, android, ios, console, windows, linux, etc, etc)
  • Which language should be used? (Python, Java, JavaScript, CSharp, etc, etc)
  • Patterns followed (Procedural, Object Oriented, Functional, etc, etc)
  • How to actually implement any of the strategies discussed.

Really - there is virtually no discussion of how at all.  It only contains the “axiomatic agreements”, the “facts”... or “answers to the ‘what’ questions”.  For example,

QUESTION:

If we assign (from the top left of the board) an index of 0..8 of each cell, what are the indexes which replace these for a 90 degree CLOCKWISE rotation around the middle cell?



ANSWER:  630741852

This is simply a fact which is true regardless of language, platform, implementation detail, etc.  In pseudo code, a unit test for an implementation might look like this:

TicTacToeBoard b = new()

b.SetCells(012345678)

b.Rotate()

Assert: b.CellOrder() == 630741852

How to rotate the cells is entirely up to each implementation.  The resulting order is non-negotiable however.

So this “specification” documents what the result should be - not how to get there.  What will change?  The different levels of players, ever word, color, symbol that shows up in the UI, strategies, patterns, translations.  All of it is subject to change, at any time.  Expect more.  

For example:

To start with there are Rotate and Flip translations, but the pattern easily allows for us to add an AntiFlip and an AnitRotate which would rotate and flip “the other way” around.

What this specification says…

This specification describes what your solution needs to do.  It does not describe how to do it.  The specification details are split up into the following primary areas of interest.

Cell Patterns            (Edit)

Many of the strategies use these patterns (edit).  These patterns use translations to limit the number of patterns which must be represented.  These translations are Rotate and Flip they are described in more details in the translation list (edit).

The patterns are all expressed as a sequence of  ##;##;##;...  



where the first # is the CellIndex and the second # is the CellState Id, always followed by a semi-colon.

The Translations is a similar format, but the ##’s refer to the number of translations, followed by which translation to use.

Target Platforms       (Edit)

This game is specified in such a way that it can be implemented on many different platforms.  The goal however, is to have as much as possible come from this specification - shared across all the platforms.  This way, when the specification changes, the code changes - across all the platforms.



This challenge is essentially about adding your solution to this list of target platforms which implement this specification.
